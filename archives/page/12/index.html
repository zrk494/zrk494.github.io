操作系统---死锁

发表于 2021-06-07 更新于 2022-12-01 分类于 操作系统

死锁

概述

  • 并发的进程互相等待对方手里占用的资源，导致各个进程阻塞，无法推进

    • 与饥饿的区别：饥饿是由于长时间等不到想要的资源而导致无法进行推进

    • 与死循环的区别（死循环是应用程序员应该解决的问题）

  • 死锁的4个必要条件

    • 一个资源只能由一个进程占用（互斥）

    • 每个进程都占用资源，且会向其他被其他进程占用的资源发送请求

    • 资源不会被抢占，只能有进程主动释放这个资源

    • 循环等待，P0请求的资源被P1占用，P1请求的资源被P2占用…..Pn请求的资源被P0占用

  • 何时发生死锁

    • 对系统资源（不可被剥夺）的进程

    • 进程推进顺序不恰当

    • 信号量使用不当

死锁处理的方法

预防死锁
  • 总体思路：确保一个必要条件不成立

  • 解决互斥的问题

    • 将互斥使用的资源改造成可以共享使用（如SPOOLing技术，是一种逻辑上的改造）
    • 至少设置一个非共享的资源

  • 解决占有并等待的问题

    • 进程运行之前，申请其所需要的所有的资源（如果不满足要求，则不运行）
  • 解决非抢占的问题
    • 方法1：申请的资源得不到满足，则立即释放资源
    • 方法2：申请的资源被其他进程占用，有由操作系统协助剥夺（按照优先级）
  • 解决循环等待问题
    • 引入ID标识每一个资源，每个资源按ID递增顺序请求资源（同类资源即相同ID的资源就一次申请完）
避免死锁

  • 安全序列：系统按照这样的序列分配资源，所有进程都可以顺利的完成（可能由多个），此时系统是安全状态，不可能发生死锁

    • 系统分配完资源之后，找不到任何一个安全序列，进入不安全状态，此时有可能发送死锁

  • 避免死锁，是通过动态的算法控制进程如何去申请资源（根据资源的附加信息块决定是否请求该资源），从而规避死锁现象

  • 银行家算法

    • 思路是提前判断某次分配是否会导致系统进入不安全状态，以此决定是否分配该资源，主要包括两部分：安全性算法+资源请求算法

    • 定义数据结构（4个部分）

      • 可分配的资源数：Available[j]=k资源Rj可用的资源数是k个
      • 最大需求数：Max[i,j]=k进程Pi请求的资源Rj有k个实例
      • 占用的资源数：Allocation[i,j]=k
      • 完成任务可能还需要的资源数：Need[i,j]=k

    • 两个重要的组成部分

      • 安全性算法

      • 资源请求算法

    • 找安全序列可以用表的形式快速手算

      • 遍历一次所有进程的资源分配信息，找出need>available的进程放入安全序列，之后这些进程释放，归还资源，改变available的值，再进行遍历，直到所有进程释放

检测死锁并解除

  • 为了方便研究这个问题，引入资源分配图（一种数据结构）

    • 边集合，包括两种类型的边，申请边（P——->R，）和分配边（R———>P）

    • 顶点集合，包括2部分，资源结点的集合（R）和进程结点的集合（P）

    • 资源分配图实例

  • 死锁检测算法

    • 一次消除不阻塞进程（指申请的资源还足够的进程）相连的变，直到无边可消

    • 若资源分配图不能简化或者不能完全简化，剩余的部分就是会发生死锁的进程

  • 解除死锁的方法

    • 资源剥夺
    • 进程撤销
    • 进程回退
