计算机网络---应用层

发表于 2021-03-29 更新于 2021-05-09

应用层

1. 应用层协议原理

1.1 体系结构（了解）

C/S (客户机/服务器)

服务器对外提供服务（不间断，一般有大量的服务器），客户机使用服务器提供的服务，不会和其他客户机直接的通信

P2P (点对点)

端系统或者节点之间直接通信

C/S+P2P（混合）

1.2 网络应用进程通信（进程的定义参照操作系统）

1.2.1 过程（一般为不同主机之间）

通过网络交换报文进行通信，包括发送进程（向网络中发送报文）和接受进程（回送报文进行响应）

1.2.2 套接字（socket，又名编程接口，api）

应用层和运输层之间（进程与计算机网络之间）的接口，在创建一个因特网应用之后，通过套接字选择一个运输层协议（UDP或TCP），还可以设置一些相关参数。

1.2.3 进程寻址*(如何去寻找另一台主机上的进程)

流程：找主机（IP地址标识是哪一个主机）—>找进程（用端口号–port number–标识）

1.2.4 用户和网络应用之间的接口

1.3 应用层协议（RFC文档中有各种协议的详细说明）

1.3.1 内容

  • 交换的报文（消息）类型（请求还是响应）
  • 报文（消息）类型的语法（格式）
  • 字段的语义
  • 进程什么时侯发送报文
  • 进程怎么发送报文
  • 进程对报文进行响应的规划

1.3.2 应用层使用的服务（服务需求）

  • 可靠数据传输（某些应用可以允许丢失一部分数据，某些必须100%可靠）
  • 吞吐量（带宽）
  • 定时（时延）
  • 安全性

1.3.3 Internet的传输服务（TCP，UDP，详见传输层）

2. HTTP（超文本传输协议，定义了报文格式和交换报文的方式）

2.1 相关概念解释

2.1.1 web页

若干对象组成的一个界面

2.1.2 对象

像是HTML，JPEG，Java程序之类的文件，地址用URL来表示

2.1.3 URL

统一资源定位符，组成包括两部分，主机名+路径名

2.1.4 浏览器

web应用的用户代理（客户机端）

2.1.5 web服务器

存储微博对象的地方（服务器端）

2.1.6 HTTP的工作

定义了Web客户向web服务器请求web页面的方式，以及服务器向客户传送web页面的方式

补充

  • HTTP使用C/S架构
    • 客户（browser），请求接收展示web对象
    • 服务器（web server），响应请求，发送对象

2.2 HTTP的工作过程

服务器在80端口等待请求—->建立TCP连接（创建套接字）—->交换报文—->关闭连接

补充说明：

  • 使用TCP作为运输协议
  • 服务器向客户发送被请求的文件，但不存储任何关于该客户的状态信息（无状态协议）

2.3 HTTP连接方式

2.3.1 非持续性连接（一个TCP连接只传送一个web对象）

过程（在浏览器输入URL之后，按时间顺序进行）

  1. 客户端向HTTP服务器进程（80端口）发起TCP连接请求 ——->
  2. 服务器在80端口等待TCP连接请求，接受连接之后向客户端发出通知 <——-
  3. 客户机下将HTTP请求消息（请求报文）（包括URL，表名客户端需要的对象）通过TCP的套接字发出 —–>
  4. 服务器收到请求消息（请求报文）之后将其解析，将客户端请求的对象封装进响应消息，通过套接字发回给客户端 <=====
  5. 服务器关闭TCP连接
  6. 客户端收到消息之后解析，并且将收到的对象显示出来（只能显示一个对象，如果还需要其他对象就继续重复上面的步骤）

性能分析

  • 响应时间

    获得一个对象的时间2RTT+文件发送时间

2.3.2 持续性连接（一个TCP连接传送多个web对象）

过程 1-4同非持续性连接，5不断开TCP连接,后续仍然通过这个连接发送HTTP消息

  • 无流水机制（客户段收到前一个响应后，再发请求）
  • 带流水机制（客户端可以不接收响应就尽快发出下一个请求）

2.4 报文格式

2.4.1 HTTP请求报文（ASCII书写）

请求行（方法字段+URL字段+HTTP版本）+首部行（可扩展）+消息体（Entity Body）

补充：

  • 首部包括多个内容，如host，user-agent，connection等，可以根据单词推测意义，格式为

    header field name: value（是可扩展的）

  • 方法字段

    • post方法
    • head方法
    • get方法
    • put方法
    • delete方法

2.4.2 HTTP响应报文

状态行（版本+状态码+短语）+首部行（可扩展）+实体体

补充：

  • 首部行中Last-Modified和Date的区别
  • 实体体中包含客户端请求的对象本体
  • 状态码，与短语相对应，指示出了请求的结果
    • 200 OK
    • 301 Moved Permanently
    • 400 Bad Request
    • 404 Not Found
    • 505 HTTP Version Not Support

2.5 cookie技术

2.5.1 定义

客户端保存用户信息（用户的访问记录）的一种机制，它允许站点对用户进行追踪，为了辨别用户身份，架设在HTTP之上

2.5.2 四个组件

  • HTTP响应报文的cookie头部行
  • HTTP请求报文的cookie头部行
  • 保存在客户端主机上的cookie文件，有浏览器管理
  • web服务器的后台数据库

2.5.3工作过程

客户端第一次访问向服务器发送连接请求，服务器创建一个唯一的id（会存入web服务器后台数据库），并将id封装在响应报文中传给客户端，客户端会将id存入本地的cookie文件，此后该服务器就可以通过id辨识该用户。

补充：session相关（日后再慢慢补充吧）

2.6 web缓存技术

2.6.1 功能（性能优化）

在不访问服务器的前提下满足客户端的HTTP请求

可以缩短用户请求的响应时间，减少流量

2.6.2 web缓存器（代理服务器）

存在于客户和服务器之间

浏览器向代理服务器发送所有HTTP请求（若没有请求的对象，则由代理服务器向服务器请求该对象并返回给客户端，同时保存该对象）

一般由组织（ISP网络服务提供商）建立

4. DNS（Domain Name System采用UDP传输协议）

4.1 DNS提供的服务(解决Internet上主机或路由器的识别问题)

4.1.1 标识主机的两种方式

  • 主机名 如www.google.com之类的便于人记忆的名称
  • IP地址 一串数字，便于路由器处理，如192.165.153.11（自己瞎写的，详细介绍看网络层相关）

4.1.2 DNS（域名系统）的地址转换服务

  • 功能是实现主机名到IP地址转换的系统（翻译）

  • 是一个DNS服务器实现分布式数据库(它允许主机查询分布式数据库的应用层协议)

  • 通常被其他应用层协议使用

具体过程：

用户主机运行着NDS应用的客户端—–>浏览器解析URL之中的主机名，并发送给客户端——->客户端向DNS服务器发送请求（包括主机名）———>客户端收到含主机IP地址的回答报文———->浏览器向该IP地址的HTTP服务器发起TCP连接

4.1.3 主机名转换为IP地址之外的其他服务

  • 主机别名

  • 邮件服务器名

  • 负载分配（负载均衡）

4.2 工作原理相关

4.2.1 不使用集中式设计（不好）

存在的问题

  • 单点失败（一旦应该服务器坏了，互联网就崩了）
  • 流量问题
  • 距离问题
  • 维护性问题

4.2.2 分布式，层次数据库

  • 三种类型的DNS服务器（总体结构是树）

    • 根DNS服务器（最上层） 全球共13个
    • 顶级域DNS服务器（第二层） 负责com, org, net, edu等顶级域名和国家级顶级域名（cn，uk等）
    • 权威DNS服务器（最底层） 组织内部的域名解析服务器

  • 客户要查询IP时的过程（例，www.amazon.com）

    • 查询根服务器，找com域名（客户端第一次查询

    • 到第二层的com域名解析服务器，找amazon.com（客户端第二次查询

    • 到第三层amazon.com域名解析器，获取IP地址（客户端第三次查询

    • 上层包括了下层的地址，用来告诉客户端下一次查询去那里

  • 本地DNS服务器

    • 在一个组织内起代理作用，相关补充如下
      • 会接收组织内所有用户发来的查询请求，做代理帮助用户查IP
      • 存在缓存，要缓存域名到IP之间的映射（解析之后立刻完成），一般缓存顶级域服务器的映射

  • 加上本地DNS之后的访问过程

    • 迭代查询

      1. 先向本地DNS服务器发请求

      2. 本地DNS作为代理

  • 递归查询

4.3 DNS记录和报文

4.3.1 资源记录（四元组）

主机名称到IP的映射，存储DNS服务器中，包括的字段：ttl，value，name，type

  • Type=A的情况
    • Name为主机的域名，value为IP地址
  • Type=NS的情况
    • Name为一个域，value为该权威域名解析服务器的主机域名
  • Type=CNAME的情况
    • name为某一真实域名的别名
    • value是真实的域名
  • Type=MX的情况
    • value是name对应的邮件服务器

4.3.2 DNS报文

  • 语义
    • 首部区域
    • 问题区域
    • 回答区域
    • 权威区域
    • 附加区域

6. 套接字编程（在用户的机器上开发）

6.1 演示套接字编程的程序的过程

客户发送一串字符到服务器—–>服务器接收到数据之后将字符转换为大写——>服务器将修改后的数据发送给客户—–>客户接受到后在显示器上显示该行数据

6.2 UDP实现

6.2.1抽象

不需要建立连接，封装信息后生成分组再发送

6.2.2 应用程序

产生输入流，建立套接字——>输入—–>封装——->发送

6.3 TCP实现

6.3.1 抽象的过程

先建立连接（1. 创建本地套接字，指定服务器进程的地址—–IP+端口 2. 服务器接到请求之后，建立自己的套接字，之后建立TCP连接），再传输数据（通过TCP连接建立的管道，进行可靠的数据传输）

6.3.2 在应用程序中

产生输入输出流，建立两边的套接字—–>输入—–>发送——>接收—–>显示
